Review the documentation at $ARGUMENTS for clarity, accuracy, and completeness.

**Accuracy**
- Do code examples actually work? Trace through them manually if short enough.
- Do described behaviours match the current implementation?
- Are version numbers, command names, and file paths correct?

**Clarity**
- Is the intended audience clear from the opening paragraph?
- Are concepts explained before they are used?
- Is jargon defined or avoided?
- Are sentences concise — no unnecessary padding?

**Completeness**
- Does a getting-started reader have everything they need to succeed?
- Are error cases or common failure modes documented?
- Are configuration options fully described (type, default, effect)?
- Are there missing examples where prose alone is hard to follow?

**Structure**
- Is the heading hierarchy logical?
- Can a reader scanning headings find what they need?
- Are related topics grouped together?

Report findings grouped as: blocking (incorrect or missing critical information) and suggestions (clarity or completeness improvements). 

End with a one-line summary: PASS, PASS WITH SUGGESTIONS, or NEEDS WORK.
