Agentic Workspace Orchestration TLDR

One-page version of the LOS filesystem orchestration idea · May 2026

Core idea

Start agents at the shared los-workspace/ root. Keep canonical repos under projects/, ticket state in visible features/ folders, repo work in isolated sibling worktrees/, and use orc to show the next step instead of making each agent reconstruct context.

Responsibilities

PartOwns
features/Durable ticket context and cross-repo handoff state.
STATE.yamlCurrent stage, owner, next action, worktree paths, runtime status.
workflows/Reusable process docs: workon, Django-to-QA, PR repair, QA automation.
workflows/commands.yamlPortable slash-style command map, such as /workon, /qa-plan, and /pr-repair.
workers/Who can do work: product, model, stages, launch template, permission boundaries.
projects/Canonical long-lived repo checkouts.
worktrees/Isolated ticket checkouts for code/test execution, kept beside projects/.
tmuxLive per-feature runtime sessions and waiting interactive prompts.
terminalBaseline fallback when tmux is unavailable; orc next prints commands to run manually.
orcStatus, next-step dispatch, template rendering, state updates, tmux helpers.

Worker files

Workers are Markdown files with frontmatter. The frontmatter is for orc; the body is for humans and agents. 

---
id: bob-developer
name: Bob the Developer
product: codex
model: gpt-5.5
reasoning_effort: high
cost_tier: medium
workflows:
  - django-feature-to-qa-automation
  - qa-automation
stages:
  - django_implementation
  - qa_implementation
  - pr_repair
default_tmux_window: app-codex
launch_mode: foreground
---

# Bob the Developer

Best for code edits, tests, debugging, Playwright automation, and PR repair.
Ask before dependency installs, destructive Git, external writes, or background execution.

Worker files can define cost_tier, model, and thinking/reasoning level so orc can recommend the lowest-cost capable worker and escalate only when needed. For pay-as-you-go model usage, consistently routing routine work to cheaper workers could plausibly save tens of percent versus using a flagship/high-thinking worker for every step.

STATE.yaml

Every worker must update STATE.yaml before stopping when status, stage, owner, next action, outputs, worktree paths, tmux runtime, blockers, or human-attention state changes. 

ticket: FLYWL-12345
slug: FLYWL-12345-some-feature
status: in_progress

stage:
  current: qa_plan
  owner: fred-documentor
  workflow: django-feature-to-qa-automation

next_action:
  worker: fred-documentor
  prompt: Continue QA planning from this feature context.
  cwd: .

runtime:
  tmux:
    session: FLYWL-12345
    detached: true
status: waiting_for_human
needs_human:
  window: qa-codex
  reason: command approval required
  since: 2026-05-30T10:14:00-05:00

orc

Use orc as the user-facing CLI name, short for orchestrator. Build the real version in Go with Bubble Tea: static binary, good process handling, and a polished TUI. Keep policy in files; keep Go focused on parsing, display, state updates, tmux, and worktree mechanics.

orc · workspace orchestrator
orc status
orc init --workspace ~/Desktop/los-workspace --with-los-defaults
orc show FLYWL-12345
orc next FLYWL-12345
orc command workon FLYWL-12345
orc advance FLYWL-12345 qa_plan
orc tmux create FLYWL-12345
orc tmux attach FLYWL-12345

First version should be read-only: status, show, and next. init can create the starter folders and templates. Add mutation, worktree creation, tmux helpers, and execution only after the state schema settles.

tmux

Each feature can have a detached tmux session: 

tmux new-session -d -s FLYWL-12345 -c /Users/cengebretson/Desktop/los-workspace
tmux attach -t FLYWL-12345

Use windows like state, app, qa, claude, servers, tests, and logs. If an agent needs human input, the tmux window waits; the human attaches, answers, and detaches. Without tmux, runtime.mode: terminal is the fallback and orc next prints the command to run in a normal shell.

Permission model

RULES.md should require approval before destructive, external, costly, security-sensitive, or hard-to-undo actions. Repo-level rules may add stricter requirements, but may not weaken workspace-level requirements.

Adoption order

  1. Create files and templates.
  2. Implement orc init to create the starter workspace.
  3. Add STATE.yaml to feature context packs.
  4. Add read-only orc.
  5. Add worker Markdown files.
  6. Add worktree and tmux helpers.
  7. Add optional execution only after logging and recovery are solid.
