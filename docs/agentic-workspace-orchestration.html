Core idea

The workspace should carry feature context across repos instead of forcing each agent session to reconstruct history from Jira, pull requests, comments, and memory. The unit of work is the ticket, not the repository.

Recommendation

Start agents at the shared los-workspace/ root. Keep canonical repos under projects/, ticket worktrees under sibling worktrees/, and feature context under features/. Commands still run with the selected repo or worktree as cwd.

flowchart LR A["Jira ticket"] --> B["features/<ticket>/"] B --> C["Django implementation"] C --> D["Django PR + repair"] D --> E["QA handoff"] E --> F["QA automation"] F --> G["Evidence + PR/Jira update"]

This is a practical version of filesystem-based agent orchestration: markdown files define context, routing, tools, rules, workflow contracts, and handoff artifacts. Scripts and tools do mechanical work; agents do interpretation, planning, implementation, review, and triage.

The shared workspace root should be visible and boring. Avoid putting durable ticket context in hidden dotfolders or inside one product repo.

Folder or filePurpose
features/ Cross-repo ticket context packs. This is the durable thread across development, QA, evidence, and handoff.
workflows/ Reusable process docs. These describe how work should proceed, but do not store live ticket state.
projects/ Canonical long-lived repo checkouts. These are the main source repos, not disposable ticket branches.
worktrees/ Isolated ticket checkouts. Worktrees can live anywhere; keeping them beside projects/ separates disposable execution surfaces from main repos.
user-overrides/ Local, gitignored preferences for tool choices, aliases, or paths. Overrides may specialize but should not weaken shared rules.

# AGENTS.md

## Scope

This is the LOS workspace root. It owns shared routing, tool policy,
feature context, and cross-repo workflow state.

## Read First

- Read `ROUTER.md` before deciding which repo or workflow owns the task.
- Read `TOOLS.md` before choosing commands, MCP servers, skills, scripts, or apps.
- Read `RULES.md` before writing files, opening PRs, or updating external systems.

## Feature Context

For Jira-driven work, first check `features/<ticket-slug>/`.
The feature folder is the durable source of ticket context across repos.

## Repo Commands

Run repo-specific commands with the selected repo or worktree as `cwd`.
Do not run package, test, or git commands from the workspace root unless
the workflow explicitly says to.

## Overrides

After reading shared rules, check `user-overrides/$USER/` when present.
User overrides may add local preferences, but may not weaken security,
approval, secrets, compliance, or audit rules.
Instruction layering

The top-level files define workspace defaults. Repo-level instruction files can specialize commands and conventions, but should not override workspace security, secrets, approval, or audit rules.

Permission rules

RULES.md should include clear approval policy. The markdown does not replace the actual harness security model, but it gives every agent the same operating expectations before it touches shared state, external systems, or expensive commands. 

## Permission and Approval Rules

Ask before doing any action that is destructive, external, costly,
security-sensitive, or hard to undo.

### Always ask first

- Deleting files, branches, worktrees, or feature context folders.
- Rewriting Git history: rebase, force-push, reset, or amend published commits.
- Installing or upgrading dependencies.
- Running commands that modify production, staging, Jira, GitHub PRs, SharePoint, or CI state.
- Starting long-running background agents or services.
- Running broad test suites or commands expected to take more than 10 minutes.
- Reading or changing secrets, credentials, env files, auth tokens, or private config.
- Sending notifications or comments to external systems.
- Changing shared workspace rules, templates, or setup scripts.

### Usually okay without asking

- Reading files.
- Searching the repo.
- Creating or editing files inside the current ticket context.
- Running targeted local validation for files just changed.
- Creating draft plans, summaries, or proposed commands.
- Updating `STATE.yaml` to reflect work just completed, unless the update changes ownership,
  closes a ticket, or marks external status.

### Ask if unclear

If an action could surprise the human, ask before doing it.
If an action affects another person, another system, or shared state, ask before doing it.

Repo-level rules can add stricter requirements for local risk areas: 

## QA Automation Repo Permissions

Ask before:

- Running headed/browser tests against shared environments.
- Updating Jira evidence.
- Creating or closing QA automation PRs.
- Replacing screenshots, traces, videos, or evidence files for an existing ticket.
Precedence

Repo-level rules may add stricter approval requirements, but may not weaken workspace-level approval requirements.

State update rule

The top-level RULES.md should also require every agent or script to keep features/<ticket-slug>/STATE.yaml current. Without this rule, orc becomes a stale dashboard. 

## State Update Rule

Every agent or script that performs work for a feature must keep
`features/<ticket-slug>/STATE.yaml` current.

Update `STATE.yaml` whenever any of these change:

- `status`
- `stage.current`
- `stage.owner`
- `next_action`
- required or completed outputs
- active repo/worktree paths
- tmux session or window names
- human attention requirements
- blocker state
- completion state

Before ending an agent session, update `STATE.yaml` so `orc status`,
`orc next`, and `orc stuck` reflect reality.

STATE.yaml

STATE.yaml is the lightweight orchestrator. It should stay small and machine-readable. Markdown files hold the detail; state answers where the workflow is and what should happen next. 

ticket: FLYWL-0000
slug: FLYWL-0000-some-cool-feature
status: in_progress

stage:
  current: django_implementation
  owner: codex
  workflow: django-feature-to-qa-automation

repos:
  app:
    main: ../../projects/los-app-los-django
    worktree: ../../worktrees/los-app-los-django/FLYWL-0000-some-cool-feature
    branch: codex/FLYWL-0000-some-cool-feature
  qa:
    main: ../../projects/los-qa-automation
    worktree: ../../worktrees/los-qa-automation/FLYWL-0000-qa-automation
    branch: codex/FLYWL-0000-qa-automation

inputs:
  ready:
    - JIRA.md
    - SPEC.md
    - PLAN.md

outputs:
  required:
    - django/PR.md
    - django/QA_HANDOFF.md

next_action:
  product: codex
  command: Continue Django implementation
  cwd: ../../worktrees/los-app-los-django/FLYWL-0000-some-cool-feature

history:
  - at: 2026-05-30T00:00:00-05:00
    stage: workon
    owner: codex
    result: feature context created
FieldMeaning
statusOverall state: ready, in_progress, blocked, review, done.
stage.currentThe current workflow stage, such as workon, django_implementation, pr_repair, qa_plan, or qa_automation.
stage.ownerThe product or person expected to handle the current stage: codex, claude, cursor, human, or github_actions.
reposMain checkout, active worktree, and branch for each repo touched by the ticket.
next_actionThe immediate instruction for the next agent, script, or human.
historyA short audit trail. Long details belong in WORKLOG.md or run files.

Waiting, blocked, and done states

State should make human attention and blockers visible without requiring someone to inspect tmux panes or logs first.

Worker definitions

Workers describe who or what can do work. Keep them as Markdown files with small YAML frontmatter blocks. The frontmatter gives orc enough structure to match a worker to a workflow and launch it. The Markdown body gives humans and agents the nuance. 

workers/
  _template.md
  bob-the-developer.md
  fred-the-documentor.md
  human-reviewer.md

The stable id is for automation. The friendly name is for humans and dashboards.

orc does not need to encode worker behavior. It only needs to:

  1. Read STATE.yaml for the current workflow and stage.
  2. Find workers whose frontmatter includes that workflow and stage.
  3. Pick the configured owner, recommend the best match, or show choices.
  4. Render the worker's launch and prompt templates.
  5. Open or reuse the worker's tmux window if requested.
Available workers for qa_plan:
1. Fred the Documentor    claude    best for synthesis and QA handoff
2. Bob the Developer      codex     best for implementation and tests

Recommended: Fred the Documentor
Design rule

Put policy in files, not code. orc should parse, match, render, and update state. The worker files explain judgment, model preference, launch behavior, and boundaries.

The top-level RULES.md should make the cost policy explicit: 

## Cost-Aware Worker Selection

Prefer the lowest-cost worker that is allowed for the workflow/stage and capable of the task.
Escalate model, thinking level, or service tier only when the state, workflow, or human says
the complexity requires it.

Ask before using high-cost workers for routine implementation, lint fixes, or small test fixes.
Record cost-tier escalation in `STATE.yaml` history.

Cost control

Cost control should be easy for humans to understand. The point is not to pick the cheapest model every time; it is to stop using expensive models and high-thinking modes for routine work by accident.

Cost rule

Use the lowest-cost capable worker by default. Escalate model, reasoning/thinking level, or service tier only when the workflow, stage, state, or human says the complexity requires it.

Work typeTypical worker tierWhy
Lint fix, typo, small selector update low Clear local context and low ambiguity.
Targeted test repair or small Playwright update low to medium Mostly implementation and validation.
Normal feature implementation medium Needs code judgment, but usually not flagship-level reasoning.
Cross-repo planning or unclear behavior high Needs synthesis across Jira, app code, QA handoff, and risk.
Architecture, security, or major workflow design high High ambiguity and high cost of mistakes.
Urgent short-latency work Fast/Priority only when justified Latency costs more; do not make it the default.

The savings can be meaningful. As a rough planning heuristic, if routine work can use lower-cost workers and only hard planning uses expensive workers, total pay-as-you-go model cost may drop by tens of percent. In many OpenAI-style pricing mixes, routing most routine work to mini or lower-cost models can plausibly save 30-70% compared with using a flagship model for every step. 

Example routing mix:

70% routine work       -> low-cost worker / low reasoning
25% implementation     -> medium worker / medium-high reasoning
5% hard planning       -> high-cost worker / high reasoning

Expected behavior:
- cheaper default execution
- explicit escalation for hard work
- fewer accidental Fast/Priority sessions
- cost choices visible in STATE.yaml history

Workflow documents

Keep workflow documents named by what they are for. WORKFLOW.md is enough for a first version. Split out a RUNBOOK.md only when the operational steps get long enough to deserve a separate procedure.

Document typeExamplesUse
Core workspace docs AGENTS.md, ROUTER.md, TOOLS.md, RULES.md Shared behavior, routing, tools, and constraints.
Workflow docs workflows/workon/WORKFLOW.md, workflows/qa-automation/WORKFLOW.md Reusable process for repeated work.
Context templates features/_template/JIRA.md, PLAN.md, QA_HANDOFF.md Copied into each feature folder.
Handoff docs django/QA_HANDOFF.md, qa/SOURCE_CONTEXT.md, qa/QA_RESULT.md Carry state from one stage or repo to another.
Run logs runs/<timestamp>/RUN.md, screenshots, traces, videos, logs Record what happened in a specific execution.

Workon belongs at the workspace level

Once feature context lives in features/ at the workspace root, workon should be a workspace intake workflow, not a Django-only repo skill. 

/workon FLYWL-0000

1. Fetch or refresh Jira context.
2. Create or update features/FLYWL-0000-some-cool-feature/.
3. Write STATE.yaml, JIRA.md, SPEC.md, PLAN.md, WORKLOG.md, DECISIONS.md.
4. Route the ticket to the first owning repo and workflow.
5. Create or select the relevant worktree.
6. Tell the agent where to run commands.

Workflow commands

Workflows can be exposed as slash-style commands, even if every agent product handles native slash commands differently. Keep a portable command map in the workspace and let orc resolve it. 

/workon FLYWL-12345
/qa-plan FLYWL-12345
/qa-implement FLYWL-12345
/pr-repair FLYWL-12345
/evidence FLYWL-12345
# workflows/commands.yaml
commands:
  workon:
    description: Create or refresh feature context from Jira.
    workflow: django-feature-to-qa-automation
    stage: intake
    worker: fred-documentor

  qa-plan:
    description: Produce QA automation plan from app handoff.
    workflow: django-feature-to-qa-automation
    stage: qa
    worker: fred-documentor

  qa-implement:
    description: Implement Playwright automation from the QA plan.
    workflow: qa-automation
    stage: implementation
    worker: bob-developer

  pr-repair:
    description: Repair CI, review, or Copilot feedback for an open PR.
    workflow: django-feature-to-qa-automation
    stage: pr
    worker: bob-developer

The portable implementation is: 

orc command workon FLYWL-12345
orc command qa-plan FLYWL-12345
orc command pr-repair FLYWL-12345

In an agent chat, /workon FLYWL-12345 can be a convention that means: read workflows/commands.yaml, resolve the command, then follow the mapped workflow and stage. If a product supports native slash commands, those native commands can call or mirror orc command.

Portability

Do not make the workflow system depend on one product's slash-command implementation. Store command intent in files; let native commands be adapters.

Worktrees

Worktrees are the natural execution layer. They can exist outside the source repo, so the workspace can centralize ticket branches while keeping the main checkouts clean.

## Worktree Location

All agent-created Git worktrees should live under the workspace-level `worktrees/` folder.

Use this shape:


Do not create ad hoc worktrees inside product repos unless a repo-specific rule explicitly

tmux runtime sessions

tmux fits as the live runtime layer. The filesystem remains the source of truth; tmux keeps the long-running shells, agents, servers, watchers, and logs organized per feature. 

features/FLYWL-12345/STATE.yaml   # durable state
tmux session FLYWL-12345           # live runtime
worktrees/...                      # isolated repo checkouts

Each feature can get a named detached tmux session: 

tmux new-session -d -s FLYWL-12345 -c /Users/cengebretson/Desktop/los-workspace

A useful default window layout:

WindowPurposeTypical cwd
state Show current state, next action, and workflow notes. los-workspace/
app Django implementation or repair work. worktrees/los-app-los-django/<ticket>/
qa Playwright automation work. worktrees/los-qa-automation/<ticket>-qa/
claude Planning, review, synthesis, or handoff drafting. los-workspace/
servers Django, Vite, Docker, or local service processes. Selected repo or worktree
tests Playwright, pytest, CI reproduction, or watchers. Selected repo or worktree
logs App logs, test logs, or PR/check watchers. Selected repo or worktree 
tmux new-window -t FLYWL-12345 -n state -c /Users/cengebretson/Desktop/los-workspace \
  'orc show FLYWL-12345; exec $SHELL'

tmux new-window -t FLYWL-12345 -n app \
  -c /Users/cengebretson/Desktop/los-workspace/worktrees/los-app-los-django/FLYWL-12345 \
  'exec $SHELL'

tmux new-window -t FLYWL-12345 -n qa \
  -c /Users/cengebretson/Desktop/los-workspace/worktrees/los-qa-automation/FLYWL-12345-qa \
  'exec $SHELL'

Attach only when needed: 

tmux attach -t FLYWL-12345

Detach with Ctrl-b d. Detached creation should be the default so starting a feature workspace does not take over the current terminal.

STATE.yaml can record the session name and intended windows: 

runtime:
  tmux:
    session: FLYWL-12345
    detached: true
    windows:
      state:
        cwd: .
        command: orc show FLYWL-12345
      app:
        cwd: worktrees/los-app-los-django/FLYWL-12345
      qa:
        cwd: worktrees/los-qa-automation/FLYWL-12345-qa

Possible orc commands: 

orc tmux create FLYWL-12345
orc tmux attach FLYWL-12345
orc tmux list
orc tmux kill FLYWL-12345
Boundary

STATE.yaml says what should happen. tmux keeps the live shells where it happens. orc connects them.

Human interaction in tmux

Treat tmux windows as persistent interactive terminals, not unattended background jobs. If an agent needs command approval, clarification, login, MFA, or review input, the window should stay open and wait for a human. 

orc tmux attach FLYWL-12345
app-codex     waiting for command approval
qa-codex      test running
claude        waiting for clarification
servers       running

The human attaches, answers the prompt in the relevant window, then detaches with Ctrl-b d. This preserves the exact prompt and terminal state instead of turning the interaction into a detached log-diving problem.

STATE.yaml can expose that a feature needs attention: 

status: waiting_for_human

needs_human:
  reason: command approval required
  window: app-codex
  prompt: Approve running Django test setup?
  since: 2026-05-30T10:14:00-05:00
Ticket        Status              Window      Reason
FLYWL-12345   waiting_for_human   app-codex   command approval required

Later, orc watch or notification integrations can surface waiting sessions. The first version only needs to show the status and make attaching easy.

Without tmux

tmux should be a first-class runtime because it gives each feature a persistent, attachable session with organized windows. It should not be required. The baseline runtime is a normal foreground terminal. 

runtime:
  mode: terminal

In terminal mode, orc next prints the recommended command and working directory: 

Next worker: Bob the Developer
cwd: /Users/cengebretson/Desktop/los-workspace/worktrees/los-qa-automation/FLYWL-12345-qa

Run:
  codex --cd /Users/cengebretson/Desktop/los-workspace/worktrees/los-qa-automation/FLYWL-12345-qa \
    "Continue from features/FLYWL-12345-some-feature/STATE.yaml"

The human runs that command in a regular shell. If the agent needs input, the prompt appears in that shell instead of a tmux window. 

status: waiting_for_human

needs_human:
  reason: command approval required
  runtime: terminal
  prompt: Approve running Playwright against staging?
  since: 2026-05-30T10:14:00-05:00

orc init can detect whether tmux is installed. If it is missing, use terminal mode by default and make tmux commands explain the fallback. 

orc tmux create FLYWL-12345

tmux is not installed.
Use `orc next FLYWL-12345` and run the printed command manually.

Multi-product orchestration

If Codex, Claude, Cursor, GitHub Actions, and Jira automation all participate, a shared state contract becomes necessary. The orchestrator does not need to be smart; it needs to be legible.

ProductGood fitReads/writes

Onboarding setup

orc init should own workspace bootstrapping. That keeps setup inside the same tool that later reads the structure, validates state, renders next actions, and manages tmux/worktree helpers.

ModeBehavior

Init safety

orc init should be conservative: do not overwrite without --force, support --dry-run, create a sensible .gitignore, and never move, symlink, or modify existing repos without explicit confirmation.

The preferred implementation path is Go with Bubble Tea. A shell or Python script is fine for a quick proof of concept, but Go gives a simple static binary, straightforward process management, good filesystem support, and a polished TUI ecosystem.

The user-facing command should be orc, short for orchestrator. It is short enough to type frequently and memorable enough for daily use. 

orc init
orc status
orc next FLYWL-12345
orc command workon FLYWL-12345
orc tui
orc tmux attach FLYWL-12345

A small banner can appear for interactive commands such as orc tui, orc init, and orc --version. Suppress it for scriptable commands such as orc status --json, when stdout is not a TTY, or when --no-banner is set.

Keep the Go app thin: parse files, validate state, render commands, update STATE.yaml, and manage tmux/worktree mechanics. Do not hard-code workflow policy in Go when it can live in workflows/, workers/, or RULES.md.

orc boundary

orc should start as a dispatcher and visibility tool, not a black-box agent launcher. Its first job is to read state and tell the human what should happen next. 

orc next FLYWL-12345
Ticket: FLYWL-12345
Current stage: qa_plan
Recommended agent: claude
Reason: planning/synthesis stage; needs Django handoff + Jira + acceptance criteria
Workspace: /Users/cengebretson/Desktop/los-workspace
Inputs ready:
  - JIRA.md
  - django/QA_HANDOFF.md
  - django/PR.md

Run:
  claude --add-dir . "Continue FLYWL-12345 using features/FLYWL-12345-some-feature/STATE.yaml"
Command typeExamplesWhen to add
Read-only status, show, next First. These commands build trust in the state schema without changing anything.
State mutation advance, block, done After the team agrees on stage names, statuses, and required outputs.
Workspace setup workon, create-worktree After repo-specific worktree rules are encoded safely.
Execution run-next, run-next --background Last. Launching agents should wait until logging, approvals, and state updates are reliable.
Foreground by default

Agent commands should run in the foreground at first. Background execution hides approvals, failures, prompts, and partial state unless the workflow has strong logging and recovery.

MVP phases

The state schema should mature before the tool becomes clever. Build in phases so the filesystem contract stays understandable.

PhaseScopeDone when
1. Files only Create features/_template/, STATE.yaml, and core workflow docs. A ticket can be represented clearly without any custom tooling.
2. Read-only dashboard Add orc status, show, and next. The tool can explain current state and recommended next steps.
3. State mutation Add controlled updates: advance, block, done. State changes are consistent and history is recorded.
4. Worktree support Create or locate ticket worktrees and write paths back to STATE.yaml. Agents can run commands in the right checkout without guessing.
5. Optional execution Add run-next for foreground execution, then optional background execution. Runs are logged, recoverable, and reflected back into state.

Adoption path

  1. Create the top-level features/, workflows/, and instruction files.
  2. Move workon conceptually to the workspace level.
  3. Add STATE.yaml to new feature context packs.
  4. Route Django work into a Django worktree using the repo setup flow.
  5. Route QA work into a QA automation worktree and read the Django handoff first.
  6. Add orc status before building a full TUI.
  7. Only turn workflows into reusable skills when they need scripts, assets, custom triggers, or portability beyond this workspace.
Guiding rule

Add the smallest durable instruction or state file that would prevent the next agent from needing the same explanation again.
